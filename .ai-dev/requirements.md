# AI电商客服系统 - 需求分析

## 项目背景

构建一个类似淘宝/京东智能客服的AI电商客服系统，采用多智能体架构（DeepAgents技术），实现售前咨询、产品问答、订单查询、优惠券、售后服务等全链路智能客服能力。本项目为MVP原型验证，重点验证多智能体编排架构的可行性。

## 核心需求（用户故事）

### US-1 智能意图路由
As a 用户, I want 系统自动识别我的问题类型, So that 我不需要手动选择服务类别

### US-2 售前咨询
As a 用户, I want 咨询产品推荐和比较, So that 我能做出购买决策

### US-3 产品知识问答
As a 用户, I want 了解产品的详细规格和特性, So that 我确认产品满足需求

### US-4 订单查询
As a 用户, I want 查询我的订单状态和物流信息, So that 我知道商品何时到达

### US-5 优惠券服务
As a 用户, I want 查询和验证优惠券, So that 我能享受优惠

### US-6 售后服务
As a 用户, I want 申请退款/换货/维修, So that 我的售后问题得到解决

### US-7 Agent间协作（V2核心升级）
As a 用户, I want 在一次对话中处理跨领域问题, So that 我不需要重复描述问题
例："我上次买的手机想退货" → OrderAgent查订单 + AftersaleAgent创建售后

### US-8 人工转接
As a 用户, I want 在AI无法解决时转接人工客服, So that 我的复杂问题得到处理

### US-9 情感识别
As a 系统, I want 识别用户情绪, So that 愤怒用户能优先得到人工服务

### US-10 满意度评价
As a 用户, I want 对客服回答进行评价, So that 系统能持续改进

## 功能范围

### IN SCOPE（MVP必须实现）
- 智能意图路由（Router Agent + 情感分析）
- 6个专业Agent（售前/产品/订单/优惠券/售后/闲聊）
- Agent间Handoff协作（核心升级点）
- ReAct循环：多轮工具调用（核心升级点）
- 对话状态机 + 槽位填充（核心升级点）
- MCP工具层（HTTP模拟）
- Java Mock后端服务
- Gradio Web界面 + 满意度评价
- 结构化日志系统
- 会话管理（TTL + 容量控制）

### OUT OF SCOPE（后续版本）
- 用户认证/授权
- RAG知识库
- 多模态支持（图片/语音）
- 主动推送通知
- 分布式部署
- Docker容器化
- 性能压测

## 验收标准

| 需求 | 验收标准 |
|------|---------|
| US-1 | Router对6种意图分类准确率>80%，低置信度降级到闲聊 |
| US-2 | 能根据用户需求推荐产品，调用search_products+get_product_detail |
| US-3 | 能回答产品规格参数，数据来自Mock API |
| US-4 | 能查询用户订单列表和详情，包含物流状态 |
| US-5 | 能查询用户优惠券和验证优惠码 |
| US-6 | 能创建售后申请和查询售后进度，敏感操作需确认 |
| US-7 | Agent能通过Handoff转交对话，上下文自动传递 |
| US-8 | 识别到无法解决/用户要求时，提示转人工 |
| US-9 | 识别到愤怒情绪时，自动建议转人工 |
| US-10 | 每次对话结束后可评价（👍/👎） |

## 非功能需求

- 响应时间: 单次对话<5秒（含LLM调用）
- 可用性: MVP阶段不要求高可用
- 安全性: API Key不暴露，用户ID隔离
- 可观测性: 结构化日志记录路由决策、工具调用、LLM响应

## 技术约束

- LLM: 阿里云百炼 qwen-plus（DashScope API）
- Python: 3.10+
- Java: 17 + Spring Boot 3.x
- 前端: Gradio
- 不使用LangChain等重框架（MVP轻量优先）
- MCP通过HTTP模拟（MVP阶段）
